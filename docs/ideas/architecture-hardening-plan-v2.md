# Circuit Architecture Hardening — Authoritative Plan (v2)

> Supersedes `architecture-hardening-plan.md` (v1, the first within-architecture pass). v2 is the architecture-questioning re-analysis: 10 staff-architect agents judged each area sound/flawed, 47 proposals were adversarially stress-tested (3-lens panels for redesigns), all 32 first-pass `needs-scoping` findings were re-verdicted, and a completeness+over-reach critic gated the result. Generated 2026-05-29 from workflow `w30d3obyl` (38 agents, ~3.1M tokens). **Read-only — no code changed.** Corrections from the critic are applied below.

## The verdict: the architecture is sound

**No area is `flawed-redesign`.** The core abstractions are explicitly to be PRESERVED:

- the `StepExecutor` contract `(ExecutableStep, RunContext) => Promise<StepOutcome>` — a genuinely deep module behind a uniform signature;
- the fail-closed write boundaries (`TraceStore.append` re-parses with zod; `RunFileStore.writeJson` refuses untagged paths);
- catalog-as-single-source-of-truth for flows; the zod-first schema layer; the byte-identical dual-host runtime bundle.

The defects are concentrated on the **supporting type layer** and a few **fused surfaces** — not the abstractions themselves. The single highest-leverage root cause: the manifest boundary (`from-compiled-flow.ts`) receives fully-typed `CompiledStep` policy fields and **downcasts them to `unknown`**, forcing every executor to re-assert shape with unchecked `as` casts. That is information destruction at a boundary the system owns on both sides — a redesign of the *type contract*, with zero runtime-behavior change.

### Redesigns

#### `runtime-core-execution/RCX-1` — Type ExecutableStep policy fields from the strict schemas; delete executor-side casts
*redesign · risk medium · effort M · confidence high · stress-test 3/3*

- **Problem:** ExecutableStep declares check/policy/join/concurrency as `unknown` (executable-flow.ts:28,40,46,70,71) even though from-compiled-flow.ts:84,103,132,136 builds them from already-strict zod types. Six executors re-spell the shape with unchecked `as` casts plus manual guards (checkpoint.ts:63,437; fanout.ts:30,42,55,68; sub-run.ts:26; relay.ts:108). A malformed compiled sub-shape that passed CompiledFlow.parse would slip through as silent undefined, and the same shape is duplicated across executors so a fix to one leaves the others wrong.
- **Current architecture:** CompiledFlow (strict zod) -> from-compiled-flow downcasts policy-bearing fields to `unknown` on ExecutableStep -> each executor upcasts back with `as {...}` and re-validates inline.
- **Proposed architecture:** Replace each `unknown` on the per-kind ExecutableStep interfaces with the inferred schema type that from-compiled-flow already produces: VerificationStep.check/ComposeStep.check: SchemaSectionsCheck; CheckpointStep.policy: CheckpointPolicy and .check: CheckpointSelectionCheck; RelayStep.check/SubRunStep.check: ResultVerdictCheck; FanoutStep.check: FanoutAggregateCheck and .concurrency: FanoutConcurrency. Where a runtime-shaped projection differs from the schema (e.g. RunFileRef vs ReportRef), define one small typed accessor in the manifest layer (checkpointPolicy(step), fanoutJoinPolicy(step), subRunPassVerdicts(step)) that parses-or-throws once with the executor's current error string. Executors call the accessor and the inline `as` casts disappear.
- **Why better:** Restores a real (deep) abstraction at a boundary the system owns on both ends: the type that crosses the manifest boundary equals the type that left the schema, so the compiler — not six hand-written guards — enforces shape, and the validation happens once at a named seam instead of being re-derived per executor. Eliminates an entire class of silent-undefined bugs and the divergence risk of duplicated shape knowledge.
- **Behavior preservation:** Add characterization tests FIRST that feed each executor a flow with a malformed policy/check/join and assert the EXACT current message (checkpoint 'is missing checkpoint policy' at checkpoint.ts:61; fanout 'has unsupported join policy' at fanout.ts:51; sub-run check.pass handling at sub-run.ts:26-29; checkpoint 'check.allow is [...]' at checkpoint.ts:443). The typed accessors must reproduce those messages byte-for-byte. Valid flows are unaffected because from-compiled-flow already emits these exact shapes.
- **Domain-model impact:** none — observable trace/result/abort behavior, flow taxonomy, and dual-host parity are untouched; only the internal TypeScript types and the location of shape validation change.
- **Migration sketch:** 1) Lock executor error strings with characterization tests. 2) Import the inferred schema types into executable-flow.ts and replace the `unknown` fields per kind. 3) Run tsc; it will surface every `as`-cast site. 4) Replace casts with direct field access or a typed accessor in manifest/. 5) Delete the synthesized `join` field per RCX-2 in the same pass since FanoutStep.check now carries the real join policy. 6) Re-run the full runtime suite. The runtime-context-boundary source-text test does not pin these field types, so it is unaffected.
- **Supersedes first-pass findings:** RC-1

Breaks the literal module cycle. Makes shared/ reasonable in isolation. The fix mirrors a pattern the codebase already uses correctly for operator-summary projection, so it is convergent, not novel.

FIRST extend tests/unit/shared/html/*.test.ts to snapshot full projector output for build/explore/prototype fixtures, then relocate modules and re-point imports/registry wiring. Verify scripts/flows/emit.ts --check and npm run check-plugin-runtime stay green since the operator summary feeds generated surfaces.

### Restructures

#### `_macro/MACRO-1` — Break both top-level cycles by relocating the connector-resolution function and the flow-projection registries to their correct layers
*restructure · risk medium · effort M · confidence high · stress-test 1/1*

- **Problem:** The two mutual top-level cycles (runtime<->flows, shared<->flows) reported by the macro import graph are caused by exactly three precise leaks, not by a wrong engine/flows boundary. (1) runtime<->flows is created SOLELY by src/flows/prototype/writers/variant-options.ts:1-5 importing three value functions from runtime/connectors/resolver.js; every other runtime<->flows edge points the correct way (runtime depends on the flow-index; the flow-index never imports runtime). (2) shared<->flows is created by src/shared/html/* and src/shared/operator-summary-writer.ts, which are flow-aware projection registries (HTML_PROJECTORS keyed by flowId importing flows/{build,explore,prototype}/reports.js; operator-summary-writer importing flows/catalog.js) mislocated in the shared junk drawer.
- **Current architecture:** src/connectors holds connector defs (leaf on schemas). src/runtime/connectors/resolver.ts holds the layered resolution policy (explicit->role->flow->default->auto). prototype flow-content imports those resolver functions, so flows->runtime (value). Separately, shared/html/index.ts + shared/operator-summary-writer.ts are flow projection registries that import flows.content/core, so shared->flows; flows.content imports shared leaf utils (resolveRunRelative, rubric), so flows->shared. Result: two cycles.
- **Proposed architecture:** Move the resolver's pure, schema-driven selection functions (assertConnectorSelectionCompatible, resolveConnectorForGuidanceInput, resolveConnectorReference) out of runtime/connectors into the top-level src/connectors leaf (which already depends only on schemas/connector). Both runtime executors and the prototype flow-writer then import connector resolution from the leaf, below both. This deletes the runtime<->flows back-edge. Separately, move shared/html/* and operator-summary-writer.ts into the flows layer as a projection registry sibling to flows/registries (e.g. flows/registries/operator-summary/ + flows/registries/html/), since they are per-flow projectors exactly like the existing writer registries. shared then holds only role-agnostic leaf utils + domain policy and no longer imports flows.
- **Why better:** Simpler because the cycles disappear with three moves and no new abstraction: the macro graph becomes a strict DAG (cli -> {runtime, app-services} -> connectors/flow-index/shared -> schemas). It is genuinely better, not merely different, because it makes the existing intended seam ('the engine never imports a flow module directly', and 'flow content never imports the engine') structurally true instead of convention-enforced — the one violating import can no longer typecheck against runtime. It also co-locates all per-flow projection logic (html, operator-summary) with the other per-flow registries instead of splitting it across shared/ and flows/.
- **Behavior preservation:** Pure code motion. The resolver functions are pure (verified: resolver.ts imports only schemas/connector + sibling connector defs); moving them to src/connectors changes no logic. HTML_PROJECTORS dispatch-by-flowId and operator-summary projection are unchanged — only their file location and import paths move. The cli still calls writeOperatorSummary with the identical signature. No generated surface logic changes (these modules are source-side; re-emit only if a moved module is bundled — verified operator-summary-writer is cli-path, not in the runtime bundle, but run flows/emit --check to confirm byte-exactness). verify (tsc/biome/vitest/emit --check) is the gate.
- **Domain-model impact:** none. No domain term, flow taxonomy, command surface, or observable output changes; the 8-flow catalog and dual-host parity are untouched.
- **Migration sketch:** 1) Move the three pure resolution fns from src/runtime/connectors/resolver.ts to src/connectors/resolver.ts (or fold into an existing connectors file); keep any genuinely engine-stateful resolution (if any) in runtime and have it call down. 2) Repoint runtime/connectors/resolver.ts, runtime/executors/{fanout,relay}.ts, runtime/run/relay-guidance.ts, and flows/prototype/writers/variant-options.ts to the leaf. 3) Move src/shared/html/ -> src/flows/registries/html/ and src/shared/operator-summary-writer.ts + src/shared/operator-summary/ -> src/flows/registries/operator-summary/; repoint cli/circuit.ts and cli/run-output.ts. 4) Run npm run verify; if any moved file is in a runtime bundle, re-emit via scripts and confirm flows/emit --check + check-plugin-runtime stay green. Land as one slice so no intermediate red.

#### `_macro/MACRO-2` — Dissolve the shared/ junk drawer into role-named layers (leaf-utils, domain-policy) and surface the implicit application-services tier
*restructure · risk low · effort M · confidence medium · stress-test 1/1*

- **Problem:** Two macro information-hiding failures: (1) src/shared is a 33-file bucket named by usage ('shared') not by role, fusing leaf utilities (run-relative-path, json-extraction, result-path), domain policy (flow-kind-policy, fanout-join-policy, terminal-verdict, policy-envelope), and (after MACRO-1 removes the projection cluster) still-mixed concerns. A bucket named 'shared' has no statable dependency rule, so anything can be dropped in and import anything, which is precisely how the shared->flows cycle arose. (2) run-envelope, run-status, history, process-evidence are top-level peers of the engine but are imported ONLY by cli — they are an application-services tier masquerading as engine-level contexts, so the directory tree does not communicate altitude.
- **Current architecture:** Flat top level: schemas, runtime, flows, shared, connectors, cli, plus run-envelope/run-status/history/process-evidence/skill-moments/release as equal peers. shared depends on schemas + flows; the four app-service dirs depend on schemas/shared/runtime and are consumed only by cli.
- **Proposed architecture:** Two low-risk moves once MACRO-1 lands. (a) Split shared by role: keep a small leaf-utility module (path/json/result-path helpers that depend only on schemas) and group the domain-policy modules (flow-kind-policy, fanout-join-policy, terminal-verdict, policy-envelope, rubric) under a clearly-named policy area — either a new src/policy peer or under flows/ if they are flow-domain policy. The naming makes the dependency rule statable ('policy depends only on schemas + leaf-utils'). (b) Optionally gather run-envelope, run-status, history, process-evidence under a single src/app (application-services) parent so the tree shows the cli->app->engine altitude. This is a tree-shape change with no graph-edge change.
- **Why better:** Better and simpler because it replaces a bucket with no invariant by areas with statable invariants — the core complaint about junk drawers (Ousterhout: a module should have a crisp responsibility; 'shared' has none). It is the durable fix for the class of bug MACRO-1 patches one instance of: with a role-named structure, a future flow-projection helper can't silently land in 'shared' and re-create a cycle. Surfacing the app-services tier makes the macro layering self-documenting for the Anthropic-eyes goal.
- **Behavior preservation:** Pure code motion + import repointing; no logic touched. verify (tsc, biome import-order, vitest, emit/byte-exact gates) is the oracle. Note tests/contracts/architecture-boundaries.test.ts may hard-pin some current paths/source-text — any such path assertion must be updated in the SAME slice as the move (the digest flags this ratchet pattern for trace.ts; the same discipline applies to any path-pinned boundary test).
- **Domain-model impact:** none. Directory grouping and module naming only; CONTEXT.md/UBIQUITOUS_LANGUAGE terms, flow taxonomy, commands, and observable behavior are untouched. If domain-policy modules move under flows/, that is consistent with them already being flow-kind policy.
- **Migration sketch:** 1) After MACRO-1, inventory shared/ remaining files into {leaf-util, domain-policy} sets by reading each file's imports (leaf-util = imports schemas only). 2) Create the role-named area(s); move files; repoint importers (mechanical). 3) Optionally create src/app and move run-envelope/run-status/history/process-evidence under it; repoint cli. 4) Update any path-pinned boundary tests in the same commit; npm run verify must stay green. Do this as 2-3 small slices (leaf split, policy split, app tier) so each is independently revertible.

#### `connectors-hosts/CH-P1` — Single built-in connector registry keyed by EnabledConnector with compile-time exhaustiveness
*restructure · risk medium · effort M · confidence high · stress-test 1/1*

- **Problem:** Built-in connector identity (capabilities, provider, supported efforts, executable, relay-dispatch target, trace narrowing) is restated across 7 sites as open if-chains over `string` that silently return undefined for unknown names. Adding the third builtin (cursor-agent) compiled cleanly but left progress.ts:45's connectorFromTrace narrowing un-updated, so cursor-agent relay.started progress events are silently dropped at progress.ts:503-505 — a live bug. expectedProvider/supportedEfforts (resolver.ts:208-220) duplicate the provider mapping each connector also restates in its selectedXModel (claude-code.ts:118, codex.ts:197, cursor-agent.ts:58).

The effort constants (CLAUDE_CODE_SUPPORTED_EFFORTS etc.) are exported from the connector modules and re-imported by resolver.ts.
- **Proposed architecture:** Introduce one internal registry `Record<EnabledConnector, BuiltinConnectorMeta>` (in schemas/connector.ts next to BUILTIN_CONNECTOR_CAPABILITIES, or a new src/connectors/registry.ts) holding per connector: provider, supportedEfforts, executable, and (in the runtime layer, to avoid a schema->connector import) the relay function. Refactor expectedProvider/supportedEfforts to index the registry by EnabledConnector (narrowing the `string` param at the call boundary); refactor relay.ts dispatch to a registry lookup with a `satisfies Record<EnabledConnector, RelayFn>` table; replace the hand-narrowing triads in relay-guidance.ts:28-33 and progress.ts:45 with `EnabledConnector.options.includes(...)`/the registry; have each selectedXModel read its provider from the registry. Fold BUILTIN_CONNECTOR_CAPABILITIES into the same registry so capabilities and provider/effort share one entry.
- **Why better:** Collapses 7 scattered restatements into one keyed table that the compiler forces to stay exhaustive — the cursor-agent progress bug becomes structurally impossible (the narrowing site would not compile against an incomplete table). It is genuinely simpler: one place to read a connector's full identity, no duplicated provider mapping, no silent `undefined` defaults masking missing entries. This is Ousterhout deep-module thinking (a single information-rich abstraction replacing many shallow conditionals) and information hiding (the connector vocabulary stops leaking into projections and guidance).
- **Behavior preservation:** All error strings (provider/effort mismatch at resolver.ts:229-239 and the per-connector selectedXModel/assertXEffort throws) are asserted verbatim by connectors.test.ts, codex-connector-schema.test.ts, cursor-agent-connector-schema.test.ts, and prototype-variant-options-writer.test.ts; the registry must reproduce them byte-for-byte. FIRST add a characterization test asserting, for every EnabledConnector.options member, the current expectedProvider/supportedEfforts values and that relayWithResolvedConnector routes each name to the right relay fn; ADD a regression test that a cursor-agent relay.started yields a progress event (pinning the bug fix). Then refactor. The progress.ts fix is the one intentional behavior CHANGE (cursor-agent now emits the progress line it always should have); confirm with the owner it is a fix not a feature.
- **Domain-model impact:** none — the enum, capability vocabulary, and connector behavior are unchanged; this only consolidates where they are written.
- **Migration sketch:** 1) Add BuiltinConnectorMeta type + registry (provider, supportedEfforts, executable, capabilities) keyed Record<EnabledConnector,...> in schemas/connector.ts; move the effort constants in. 2) Rewrite resolver.ts expectedProvider/supportedEfforts to read the registry; narrow callers. 3) Add a runtime-side `Record<EnabledConnector, RelayFn> satisfies ...` dispatch table in relay.ts. 4) Replace relay-guidance.ts:28-33 and progress.ts:45 narrowing with registry/enum membership; fix the cursor-agent drop. 5) Have selectedXModel read provider from the registry. 6) Run characterization + new regression tests, then npm run verify.
