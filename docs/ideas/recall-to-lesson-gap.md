# Closing the Recall-to-Lesson Gap

Status: design, ready to build
Date: 2026-05-30

The first real-world exercise of Circuit's memory system was the circuit-land
landing-page build on 2026-05-30 (three Build runs: two one-second aborts, then
a complete). It produced a clean, reproducible failure that this doc diagnoses
and fixes.

The observed regression: Build run 1 aborted because the repo had no
`verify`/`test`/`check` npm script. Run 2 recalled run 1 (memory fired) and
aborted identically. Run 3 succeeded only after the operator added the script by
hand. Memory surfaced the fact of failure and never the fix.

This builds on [`self-auditing-memory.md`](./self-auditing-memory.md) (the
effectiveness-ratchet thesis) and [`project-execution-memory.md`](./project-execution-memory.md)
(the cited-fact substrate). External grounding lives in
[`docs/learnings/codebase-memory-research.md`](../learnings/codebase-memory-research.md),
which predicted exactly this recall-to-lesson gap. The diagnosis here was
verified against the real engine and the live circuit-land run corpus, and the
plan was generated by an independent design panel and adversarially judged.

## 1. The corrected root cause

The handoff framed this as "the lesson was never encoded." That framing is
wrong, and the correction shapes the whole fix. The lesson is captured in four
typed places already:

- `trace.ndjson`: the `step.aborted` reason and the `run.closed` reason
- `reports/run-envelope.json`: `completion_gate.claim_results[0].gap`
- `reports/result.json`: `reason`

All four read "Cannot choose verification commands because package.json is
missing required script one of verify, test, or check." That is the exact fix.
So this is not a capture gap. It is a retrieval gap, and three independent,
verified defects conspire to produce it.

### Defect 1: a facet bug (verified)

`src/app/history/extract.ts` `buildFacets()` (line 332) adds the `failure` facet
only when `outcome === 'aborted'`. But the engine records aborted runs with
`outcome: 'failed'` on the run-envelope document (the `RunEnvelopeOutcome`
schema is `complete | needs_attention | blocked | failed | handoff`; there is no
`aborted`). So the run-envelope doc, which is the doc that actually wins recall,
is faceted `context`, not `failure`. `memory-preview.ts` `appliesTo()` keys on
that facet, so the recalled hint was classified `context` and read as a status
line, not a lesson.

The bug is asymmetric. The `step.aborted` and `run.closed` trace documents do
get the `failure` facet (their trace kind matches), and their summary is the
gap text. But those documents lose recall to the goal-echoing envelope (defect
2), so the correctly-faceted lesson never surfaces.

### Defect 2: run-start recall queries the goal, not the failure (verified empirically)

`run-start-recall.ts` (line 84) sets `perRunLimit: 1`, and `circuit.ts` passes
the operator goal as the query. At limit 1, each prior run contributes exactly
one document, and the run-envelope wins because it echoes the operator's entire
goal and therefore scores highest on a goal-worded query. The probe below
confirms this against the live corpus:

- `circuit history query "<the run-1 goal>" --flow build --per-run-limit 1`
  returns, for each prior run, the `reports/run-envelope.json` report doc with
  summary "Run closed with outcome aborted" and no `failure` facet.
- Raising `--per-run-limit` surfaces more docs, including failure-faceted
  `result.json` docs, but every one of their summaries is still "Run closed
  with outcome aborted." None carries the gap.
- The lesson IS retrievable, just not by the goal. A query made of the fix's own
  vocabulary (`"package.json missing required script verify test check"`) returns
  three results at `--per-run-limit 1`, topped by the `step.aborted` trace docs
  whose summary is the missing-script reason, scoring well above the `score > 0`
  gate.

Correction to an earlier draft: this doc previously claimed that fix-vocabulary
query "returns zero results, filtered by the `score > 0` gate." That was wrong.
It was a measurement artifact: the probe passed `--per-run-limit 8`, which the
CLI rejects as out of range (max 5) and answers with an error envelope, not an
empty result set. Run at a valid limit the query retrieves the lesson docs fine.

So the defect is narrower than "the lesson is unreachable." It is reachable by
the right words; run-start just never queries them. Run-start uses the goal, so
the goal-echo envelope wins the single per-run-limit-1 slot and the fix-bearing
trace docs never surface. The lesson "this repo's build needs a check script" is
scoped to (project, flow), so the right retrieval relation is scope membership,
not goal-lexical overlap. Recall only fired at all in circuit-land because runs 2
and 3 reused run 1's exact goal; a different-worded build goal would have
surfaced the same useless envelope echo.

### Defect 3: the write-back and measurement loops are unwired (verified)

`src/memory/project-distill.ts` `distillProjectFacts()` already mines
`step.aborted` reasons, clusters by (flow_id, normalized reason head), redacts
the raw tail, and proposes a `prior_failure` project fact. It has zero non-test
callers. Nothing persists its proposals. The reader
(`project-injection.ts` `loadProjectFactCandidates`) is already wired into
`run-start-recall.ts` (line 105) and re-verifies staleness at injection, but it
reads an empty `.circuit/memory/` store (only `circuit memory note` populates it
by hand).

The measurement loop has the same shape. `memory-merge.v1.json` and
`memory-effect.v1.json` are written only by the manual CLI commands
(`circuit history memory-merge --write` / `memory-effect --write`). Nothing
builds them at run close. `run-start-recall` reads `memory-effect.v1.json`;
absent, earned-precision fails open with `effect_report_unavailable` and no
measured suppression. circuit-land showed exactly this.

The unifying root cause: the recall readers are wired into the run lifecycle;
the producers (lesson-bearing summaries, distilled facts, measurement reports)
are not. The self-auditing-memory thesis says capture, measure, and prune are
byproducts of Run, never commands. In the shipped code they are all still
commands.

### One correction the plan must respect

The facet fix and any extraction-summary edit, while correct, are not sufficient
alone. The doc that wins recall at `perRunLimit: 1` is the goal-echo envelope,
and `query.ts` `snippet()` centers its window on the matched terms, which are
the goal-echo words, so the gap text can stay buried below the window even after
the summary is fixed. The cheap extraction fix is a floor; the flow-scoped
write-back is the load-bearing fix.

One claim we do not build on: an early proposal wanted a "reserved failure
floor" in the recall budget, assuming distilled facts enter with `unknown`
staleness and would be crowded out. Verified false: `project-injection.ts`
`reverifyStaleness()` recomputes the source hash at injection and returns
`fresh`/`source_hash_verified` when the cited trace is unchanged, so a
freshly-distilled fact ranks at `neutral_fresh` alongside lexical hits with no
floor machinery. The plan drops the floor.

### The two-vocabulary root cause (the unifying mechanism)

Defect 1 is one symptom of a single schema inconsistency, and it is worth naming
because it ties the plan's steps together. A stopped run is recorded with two
different outcome words. `RunClosedOutcome` and `reports/result.json` admit
`aborted`; `RunEnvelopeOutcome` admits only
`complete | needs_attention | blocked | failed | handoff`, with no `aborted`. So
the same stopped run appears as `aborted` in the trace and result, and as
`failed` in the envelope. Three consequences fall out of that one split:

- `buildFacets` tests `outcome === 'aborted'`, so it misses the `failed`
  envelope and under-faceted the doc that actually wins recall (Defect 1).
- the distiller (`distillProjectFacts`) mines `aborted` from `step.aborted`
  trace entries, while the document that wins run-start recall is built from the
  `failed` envelope. The distilled lesson and the recall winner therefore come
  from different outcome vocabularies.

Step 1's `FAILURE_OUTCOMES` set closes the facet half. But the deeper through
line is that failure detection must treat the two vocabularies as synonyms
everywhere it matters (faceting, summary selection, the distiller's mining key,
and the recall ranker), or a step that fixes one surface leaves the other
mismatched. Step 3 inherits this directly: the distilled flow-scoped lesson
(mined from the `aborted` trace) must be able to out-rank the goal-echo envelope
(carrying `failed`), or the pushed lesson never surfaces. Reconciling the
outcome vocabularies, or canonicalizing them through one `isFailureOutcome`
predicate, is the cross-cutting fix that Steps 1 and 3 both depend on.

## 2. The sequenced plan

Each step is safe to ship alone, and each later step depends only on what
shipped before it.

### Step 1 (ships first): extraction and facet fix so the recalled hint speaks the fix

Changes:

- `extract.ts` `buildFacets()`: replace `input.outcome === 'aborted'` with
  membership in a module-local `FAILURE_OUTCOMES = new Set(['aborted', 'failed',
  'blocked'])`, keeping the existing `relay.failed` / `step.aborted` /
  `checkOutcome === 'fail'` clauses. Add a shared `isFailureOutcome(outcome?)`
  helper for the call sites below. Do not sweep in `needs_attention` or
  `handoff`.
- `extract.ts` `makeRunDocument()` and `makeReportDocument()`: on a failure
  outcome, prefer the diagnostic over the boilerplate close line. Run docs
  prefer the `run.closed` / `result.json` `reason`; report docs (the
  run-envelope path) prefer `body.completion_gate.claim_results[0].gap` when
  non-empty. Clip to the existing `HIGH_VALUE_TEXT_LIMIT`. Non-failure paths
  unchanged.
- `memory-preview.ts` `hintText()`: when `appliesTo()` returns `prior_failure`,
  lead with `hit.doc.summary` rather than `hit.snippet`. This defeats the
  snippet-window-buries-the-gap risk because the summary now holds the gap text
  directly.

Why first: cheapest edit, fixes whichever failure doc surfaces today, and flips
`appliesTo()` to `prior_failure`, which everything downstream relies on.

Cost low. Reversibility easy. Not a Codex decision.

Invariant it must not break: hint-only authority and cited/hash-verified
staleness (we change only summary and facet text on documents that still cite
the same hashed bytes); redaction (we surface only the typed `gap`/`reason`
fields the engine already writes for operator display, never the raw
stdout/stderr tail).

Proof: a failing-first regression in the indexer tests mirroring the circuit-land
shape (`result.json` outcome `aborted`, envelope outcome `failed`, gap text
present). Assert: envelope doc facets include `failure`; run-doc summary equals
the abort reason; envelope summary equals the completion-gate gap; the resulting
hint's `applies_to` is `prior_failure` and its text contains "missing required
script"; a `handoff`/`needs_attention` envelope is not faceted `failure`; and no
stdout/stderr substring leaks. Then re-run the live probe against the
circuit-land corpus to confirm the winning hint now carries the gap.

### Step 2 (ships next): auto-build memory-merge and memory-effect at run close

Changes:

- New `src/app/history/memory-reports-byproduct.ts`:
  `writeMemoryReportsByproduct({ repoRoot, indexDir?, now })` calls
  `buildMemoryMergeReport` then `writeMemoryMergeReport`, then
  `buildMemoryEffectReport` then `writeMemoryEffectReport` (the exact functions
  the CLI already calls). Wrap in try/catch returning warnings, never throwing.
  `buildMemoryEffectReport` rebuilds the merge in-process, so the two files
  cannot desync.
- `src/cli/post-run-artifacts.ts` `emitPostRunArtifacts()`: add a fifth,
  last-in-order artifact step `memory-reports`, wrapped in the existing
  `tryPostRunArtifact` guard, gated on `runEnvelope !== undefined`. Add
  `repoRoot` (and optional `indexDir`) to the input. Fires at all three
  `circuit.ts` emit sites.
- `recall-precision.ts` `composeIndicator()`: reword the cold-start branch so
  "effect report exists but this flow has not reached the measurement floor"
  reads as measurement-in-progress, not broken measurement. Prose only.

Why next: promoting a single-failure fact in Step 3 without a measured-non-harm
verdict is unaudited guessing. This step makes the verdict substrate exist so
Step 3's promotion is gated, not blind. Honest caveat: on the three-run corpus
the effect report reads `not_enough_data` for every arm
(`DEFAULT_MIN_ARM_SIZE = 2`), so this step alone changes substrate state
(`effect_report_available: true`), not run-2 behavior. The payoff is back-loaded
and that is acceptable for a substrate step.

Cost low. Reversibility easy. Not a Codex decision.

Invariant it must not break: byproduct-not-command (this strengthens it by
moving the build off the manual CLI), hint-only (writes only the two measurement
reports, promotes no hint), and the engine boundary (lives in `src/app`, touches
nothing in `src/runtime`).

Proof: idempotency and cold-start tests first (red). Over a fixture corpus with
at least one envelope both files appear with valid schemas, and a second
invocation produces identical content modulo `generated_at`; an empty runs base
yields warnings and a valid empty report; an integration test proves the next
run reads `effect_report_available: true`.

### Step 3 (ships later): flow-scoped write-back, distill at close and persist as a (project, flow) fact

Changes:

- New `src/memory/project-write-back.ts`:
  `recordRunCloseProjectFacts(options)` that (1) calls `distillProjectFacts`,
  (2) for each auto-recordable proposal calls `appendProjectFact` (idempotent by
  deterministic id, so re-distilling every close cannot duplicate), and (3)
  returns the announced `operator_indicator` strings. Non-auto-recordable
  proposals are returned proposed-only.
- `src/cli/circuit.ts` closed-run path (after `emitPostRunArtifacts` returns):
  one guarded call to `recordRunCloseProjectFacts`, wrapped in
  `tryPostRunArtifact('memory-distill', ...)` for fail-open. This is the
  symmetric twin of `prepareRunStartHistoryRecall` at run start: recall reads
  project facts at start, this writes them at close. Surface the returned
  indicators alongside `recallMemoryIndicator`.
- `project-distill.ts`: add an exported `DETERMINISTIC_ENVIRONMENTAL_REASONS`
  allowlist keyed on `normalizeReasonHead` output, seeded with the real
  circuit-land head. Change the gate from a module-level `minDistinctRuns` to
  per-cluster: `requiredRuns = allowlist.has(head) ? 1 : (options.minDistinctRuns ?? 2)`.
  For allowlisted heads, attach a short, code-owned remediation constant to the
  hint text ("Add one of: verify, test, or check to package.json scripts before
  the build can verify."). The remediation string is authored in-repo, never
  mined from the run.

The reader side needs no change: `loadProjectFactCandidates` is already wired at
run start, re-verifies the cited trace hash, and feeds the fact through
`applyEarnedPrecision`. A fresh trace re-verifies to `fresh`, so the fact ranks
at `neutral_fresh` next to lexical hits with no floor.

Why later: it depends on Step 2's effect report to gate promotion on measured
non-harm. It is the structural fix for defect 2, because a (project, flow) fact
is injected by flow scope, not goal overlap, so it reaches the next run through
the door goal-lexical ranking cannot reach.

Cost medium. Reversibility moderate.

Hard-to-reverse contract decision, warrants a Codex pass (AGENTS rule 6):
auto-recording a fact at run close without operator confirmation breaks the
distiller's tested propose-first default. Once runs write facts unprompted,
downstream stores and prune/forget semantics depend on that behavior. So split
Step 3:

- Step 3a: wire `recordRunCloseProjectFacts` and the run-close call, persisting
  only proposals already meeting the `>= 2` default. This closes the unwired
  loop with zero policy change and is fully revertible by deleting the helper
  and one call site.
- Step 3b: add the single-run allowlist plus remediation in propose-only mode
  (announce the indicator, do not `appendProjectFact`) until a `/codex` pass
  blesses auto-recording the typed allowlist. The allowlist entries are
  high-precision and hash-re-verified next run, so auto-recording only the
  allowlist is the defensible scope; everything else stays propose-only.

Invariant it must not break: hint-only (every persisted record is
`authority: 'hint_only'`, injected through the gate), cited/hash-verified
(proposal carries `source.sha256` and a trace ref, reader re-hashes at
injection), redaction (the allowlisted head has an empty tail, the remediation
is a code constant), byproduct-not-command (runs inside the run-close artifact
path), and the engine boundary (new logic in `src/memory` and `src/cli`, none
in `src/runtime`, no `engineFlags` entry).

Proof: keep the existing "single isolated abort yields no proposal" case for
non-allowlisted reasons. Add: a two-run abort cluster lands in `project.v1.jsonl`
at close and is injected next run (3a); one build abort with the missing-script
reason produces a persisted-or-proposed `build` fact while a non-allowlisted
single abort still produces nothing (3b); the remediation text equals the code
constant and the raw reason tail never appears in any stored hint; the persisted
fact's cited trace re-verifies fresh when unchanged and stale when mutated; a
test pins the real circuit-land head string against the allowlist entry so a
normalizer change cannot silently break it.

## 3. What Step 1 alone would have changed on circuit-land run 2

On run 2, recall queried with the goal at `perRunLimit: 1` and surfaced one doc
from run 1: the run-envelope report doc, which won because its text echoes the
goal. Today that doc has outcome `failed` (no `failure` facet) and summary "Run
closed with outcome aborted", so the operator saw a hint classified `context`
that said only the fact of failure plus the goal echoed back.

After Step 1, that same winning doc gets the `failure` facet (so `appliesTo()`
returns `prior_failure`), gets its summary set to the completion-gate gap, and,
because `hintText()` now leads with the summary on a `prior_failure`
classification, the gap can no longer be buried behind the goal echo. The hint
run 2 received would have read the missing-script diagnostic verbatim,
classified as a prior-failure lesson, cited and hash-verified to run 1's
envelope. The agent would have been told the actionable cause before
re-attempting. Step 1 makes the diagnostic legible; Step 3 makes it structurally
reliable across any future build goal in the repo, not just goals lexically
similar to "landing page."

Honest caveat on this example. The transcript of the orchestrating session shows
that circuit-land run 2 did not actually repeat because memory failed. The
orchestrator read run 1's abort reason immediately and tried to add the script,
but its package.json edit silently failed (a find-and-replace mismatch) and it
relaunched before confirming the fix had landed. The orchestrator already knew
the cause, so better memory would not have changed that specific run. Run 2 is
used here to illustrate the retrieval mechanics, not as proof that this fix would
have prevented that repeat. The fix's real payoff is the case run 2 was standing
in for: a later or context-compacted session that runs Build against this repo
with no living memory of the check-script requirement. The diagnosis in section 1
is verified against the code and the run corpus and does not depend on this
anecdote.

## 4. Honest risks and the proof each needs

- Cold-start and thin corpus. With three runs the effect report honestly reads
  `not_enough_data`, so Step 2 changes substrate state, not run-2 behavior. By
  design and disclosed. Proof: the cold-start test asserting a valid
  empty-but-honest report and the reworded indicator.
- Distractor effect (a fixed gap resurfacing). Three guards: recency ranking
  ages out a gap once the repo's newest run no longer aborts; injection-time
  hash re-verification sinks a hint whose cited trace moved or changed; and once
  Step 2's measurement matures, the earned-precision gate hard-suppresses any
  failure fact showing correlated negative effect. Proof: staleness fresh/stale
  and recency-ordering tests.
- Over-faceting (Step 1). Only `aborted`, `failed`, `blocked` join
  `FAILURE_OUTCOMES`, matching what `memory-effect.ts` and `memory-merge.ts`
  already treat as adverse. Proof: a `handoff`/`needs_attention` envelope is not
  faceted `failure`.
- Summary leak (Step 1). A `gap` or `reason` could carry a long tail. Mitigated
  by the existing text clip and a no-stdout/stderr-substring test.
- Allowlist coupling (Step 3). Keyed on exact `normalizeReasonHead` output.
  Proof: a test pins the real circuit-land head string.
- New write authority (Step 3b). The one decision worth the `/codex` pass. Ship
  3a and propose-only 3b first; do not flip auto-record until the contract,
  including prune and forget semantics, is blessed.
