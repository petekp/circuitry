# Per-step validation gate — stop-and-fix before the next step

Idea for adding an explicit verification phase to every Circuit dispatch
step, so that no step advances until its output has been checked against
declared acceptance criteria. Captured 2026-05-07 from OpenAI's
"Run long-horizon tasks with Codex" article
(https://developers.openai.com/blog/run-long-horizon-tasks-with-codex).

## The trigger

The article walks through a 25-hour autonomous Codex run that produced
~30k lines of code across 10+ feature areas. The setup is
straightforward — three markdown files (spec, plan, runbook) plus a
living documentation log — but the *load-bearing* discipline they call
out is small and specific:

> "Stop-and-fix rule: if validation fails, repair before moving on."

After every milestone the agent runs lint, typecheck, tests, and build,
and refuses to advance until those pass. They don't dress it up. It's
just: each milestone has acceptance criteria, the criteria are
machine-checkable, and "step is done" means "criteria pass," not "the
agent says so."

This is the cheapest, simplest piece of the whole article, and it's the
piece that explains how a 25-hour run doesn't end in 24 hours of
compounding rot.

## What's actually being described

Three things, in order:

1. **Acceptance criteria are declared up front, per milestone.** Not
   inferred from the prompt, not improvised by the agent. Written down
   alongside the milestone itself, before work starts.
2. **Validation runs at the milestone boundary.** Lint, typecheck,
   tests, build — whatever's relevant. The validator is *not* the same
   prompt-loop that wrote the code. It's a separate check, deterministic
   where possible.
3. **Failure blocks advancement.** No "we'll fix it next milestone."
   The current milestone repairs until validation passes, or escalates.

The trick is that all three together change the meaning of "done." Done
stops meaning "the agent produced something" and starts meaning "the
artifact passes a check that the agent didn't author."

## Why this is the right idea to borrow for Circuit

Circuit's whole value proposition is *long-horizon orchestration that
doesn't drift.* The single failure mode that breaks that promise is a
step that claims success while leaving a broken artifact behind. Every
downstream step then builds on rot, and the operator only finds out
hours later — usually at synthesis, sometimes at Close, occasionally not
until they read the output and realize what they have isn't what they
asked for.

Today, Circuit's "step is done" gate is implicit. A dispatch worker
returns; the dispatch loop accepts that return; the workflow advances.
There's no distinct verification stage between "worker returns" and
"workflow advances." If the worker is wrong about being done, the
workflow has no way to know.

A validation gate is the smallest possible change that closes that hole.
It's also a natural foundation for the bigger ideas already on the
roadmap:

- **Checkpoint/resume (FU-T12, currently deferred)** is much safer when
  every checkpoint is *known-good.* Resuming from a step whose validator
  passed is sound. Resuming from a step that "succeeded" by vibes is the
  reason checkpoint/resume is hard.
- **Long-horizon supervision** (see `long-horizon-supervision.md`) is
  about trajectory — is the run still pointed at the goal? Validation is
  about correctness — is *this* step's output sound? They're
  complementary, on different clocks. Validation fires every step;
  supervision fires every N minutes. Build validation first; supervision
  becomes much cheaper to add when each step is already verified.

Among the ideas in the article — the three-file pattern, durable
documentation, course-correction without reset, worktree isolation —
this is the only one that addresses the actual root failure mode
Circuit is shaped to solve. The others are nice; this one is structural.

## The shape that fits Circuit

The dispatch step contract grows one new field: a list of *acceptance
criteria.* The dispatch loop grows one new sub-phase between "worker
returns" and "step is marked complete": *validate.*

**Acceptance criteria, declared on the step.** Each step in a recipe
declares what it means for that step to be done. Criteria are
machine-checkable, not freeform prose. Examples in shape:

- "Schema X validates against artifact Y."
- "Command `pnpm test` exits 0."
- "Output file contains required fields A, B, C."
- "No grep match for forbidden token in changed files."

The criteria are part of the recipe, authored alongside the step. They
are not generated by the worker — that defeats the point.

**Validation sub-phase, after the worker returns.** When a worker
returns, the dispatch loop runs the criteria. Each criterion is either a
deterministic check (script, schema validate, command exit code) or, if
truly necessary, a narrow LLM-judged check with a fixed prompt and a
strict yes/no contract. Deterministic is strongly preferred; LLM-judged
is allowed only when no deterministic check is possible.

**Failure paths, declared per step.** On validation failure the loop
chooses among:

- *Retry with feedback.* Re-invoke the worker with the failed criterion
  as input ("test X failed: \<output\>; repair and re-emit"). Bounded
  retry count.
- *Escalate to operator.* Surface the failure with the criterion, the
  worker's output, and the validator's verdict. Workflow pauses.
- *Hard-fail the recipe.* Some failures are unrecoverable in-loop and
  the right answer is to stop the whole run rather than burn budget
  retrying.

Default policy is retry-then-escalate. Per-step override is allowed.

## Design constraints to get right

**Criteria must be machine-checkable.** "Tests pass" is checkable.
"Code is well-structured" is not. If a step can't be expressed in
checkable criteria, that's a signal the step itself is wrong-shaped —
either it's two steps masquerading as one, or it belongs at Review (with
a human verdict) rather than at dispatch (where the loop has to decide
autonomously). Resist the temptation to back-fill with LLM-judged
criteria for everything; that just moves the rot one level down.

**Authoring friction is the product, not a bug.** Adding criteria to
every step makes recipes harder to write. That's correct. A recipe
without criteria is a recipe that can lie to you about being done. The
right response to "this is annoying to write" is "good — now you have
to think about what done means," not "let's make criteria optional."

**The validator is not the worker.** If the same prompt-loop that wrote
the artifact also decides whether the artifact is done, you've changed
nothing. Determinism where possible. Where LLM-judged is unavoidable,
the judge runs a *separate, fixed* prompt — not the worker's prompt with
"and check your work" appended.

**Validation is not Review.** Reviewer at Close runs once with full
context and issues a verdict on the whole run. Validation runs many
times during the run with narrow context and decides "may this step
advance?" They have different scopes, different prompts, different
budgets. Don't collapse them.

**Criteria failures need a budget cap.** A step that loops forever on a
criterion it can't satisfy is worse than one that fails fast. Default
something like three retries, then escalate. Visible counter in the run
record.

**Criteria themselves can be wrong.** Sometimes the worker is right and
the criterion was authored badly. The escalation surface needs a clean
"the criterion is wrong, here's why, please fix the recipe" outcome, not
just "retry harder." This is the operator-loop on recipe authoring; it
should be cheap.

## Where this does *not* fit

**Steps with genuinely subjective output.** A step whose deliverable is
"a written summary of the situation" doesn't have clean criteria. The
honest answer is: don't pretend. Either decompose it into checkable
sub-steps, or move the verdict to Review where a human can judge.
Validation gates are not a universal acid — they're a discipline for the
steps where they fit. Other steps stay as they are.

**Tight interactive loops.** When the operator is sitting at the
keyboard and the workflow is short, *the operator is the validator.*
Adding a heavyweight per-step gate to a 90-second interactive workflow
is friction without payoff. The gate earns its keep on long, unattended
runs.

**Replacing tests in the codebase.** Validation criteria are about
"this step's output is sound," not "the system is correct overall." The
project's own test suite still runs at the appropriate seams. The
validation gate is *workflow-level*, not codebase-level.

## Honest tradeoff

The cost is real: every step now needs criteria. Some recipe authors
will resent this. Some steps don't have obvious criteria and the author
has to do real thinking to find them. Some criteria will be wrong on
first authoring and will need iteration.

The benefit is a property the system doesn't have today: *if a workflow
finishes, every step's output was actually verified.* That property is
what makes long-horizon runs trustable. Without it, "Circuit ran for 6
hours and it's done" is a hope, not a claim.

Worth it. The methodology strip cut speculative scaffolding; this is
not speculative scaffolding. This is the discipline that the cut
methodology was *trying* to encode and didn't quite land. A small,
load-bearing piece is the right shape — and "small" matters: this is
one new field on the step contract and one new sub-phase in dispatch,
not a framework.

## What to prototype

The cheapest experiment that proves the shape, not the full system.

**Step 1 — extend the step contract.** Add an optional `criteria` field
to dispatch step definitions. Type it as a list of structured checks
(start with two kinds: command-exit-code and schema-validate). Don't
wire validation yet. Confirm the recipe authoring shape feels right by
adding criteria to two or three existing recipe steps.

**Step 2 — add the validate sub-phase, deterministic only.** Between
worker-return and step-complete, run the criteria. On pass, advance. On
fail, log and hard-fail the run (no retry yet — keep it simple). Run an
existing recipe and confirm passing criteria are invisible
(no behavior change) and failing criteria stop the run cleanly.

**Step 3 — retry-with-feedback path.** When validation fails, re-invoke
the worker with the failed criterion in the input. Bounded retry count
(default 3). Confirm a deliberately-broken worker-output is repaired in
the retry loop, with the retry attempts visible in the run record.

**Step 4 — escalate path and operator override.** When retries
exhaust, the workflow pauses with a structured handoff: failed
criterion, worker output, validator verdict. Operator can: edit the
artifact and retry, edit the criterion and retry, or kill the run.
Confirm this loop with a deliberately-bad criterion (right artifact,
wrong check) and verify the "fix the criterion" path is fast.

**Step 5 — LLM-judged criteria, narrow.** Only after steps 1–4 prove
deterministic gates are valuable. Add one LLM-judged criterion type with
a fixed prompt that returns a structured yes/no + reason. Audit
generously — every LLM-judged verdict gets logged with full inputs so
the operator can spot-check that the judge is honest. If the judge is
unreliable, the answer is *fewer* LLM-judged criteria, not better
prompts.

Steps 1–2 are the minimum viable shape. Steps 3–4 are the practical
ergonomics. Step 5 is optional and should be resisted until clearly
needed.

## Open questions

- Where do criteria live in the recipe DSL? Inline on the step, or in a
  parallel file? Inline is more discoverable; parallel keeps the recipe
  narrative readable. Probably inline with a way to factor common checks
  out into reusable named criteria.
- What's the format for a criterion? A small typed object
  (`{ kind: "command", cmd: "pnpm test" }`) is unambiguous but verbose.
  A shorthand string DSL is concise but ambiguous. Start typed; add
  shorthand only if the typed form actively hurts readability.
- Should validation run in the same worker process or a separate
  isolated process? Separate is safer (a misbehaving worker can't fake a
  pass) but more expensive. Probably separate by default; allow inline
  for cheap deterministic checks like schema-validate.
- How does this interact with the existing Reviewer at Close? Reviewer
  should *see* the validation history (which criteria ran, which passed,
  which were retried) — that's signal. But Reviewer's verdict is
  separate and not blocked by per-step gates. They're independent
  layers.
- Does failure-to-validate produce a *learning artifact*? E.g. if the
  same criterion fails frequently across runs, the recipe probably
  needs editing. Worth logging but probably not worth automating yet —
  see `self-improving-circuit.md` for the broader version of this.
- How do criteria compose with the dispatch adapter choice? An
  agent-adapter step can run any deterministic check. A codex-adapter
  step is read-only sandboxed (per `MEMORY.md`) — can it run, e.g.,
  `pnpm test`? Probably yes for read-only checks; no for anything that
  writes. The criteria language has to respect adapter boundaries.
- What's the operator UX when a run pauses on validation? A single
  "approve / fix / kill" prompt is right. Anything more complex turns
  the gate into ceremony. Match the existing pause-and-resume UX as
  closely as possible.
