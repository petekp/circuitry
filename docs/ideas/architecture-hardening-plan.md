# Circuit Architecture Hardening — Analysis & Plan

> Stage 1 deliverable of the decomplect/streamline/harden Goal. **Read-only analysis — no code changed.** Generated 2026-05-29 from background workflow `wu4hnjzdf` (72 subagents, ~5.1M tokens, ~80 min).

## How this was produced

6 cross-cutting survey agents (dependency graph, complexity, duplication, dead code, test coverage, doc/contract drift) → 9 per-subsystem deep analyzers applying four lenses (decomplect / streamline / harden / boundary) → adversarial verification of every medium-or-above finding (each verifier prompted to *refute*) → plan synthesis → completeness critic.

**Hard constraints honored throughout:** product/domain model fixed (CONTEXT.md, UBIQUITOUS_LANGUAGE.md), observable behavior preserved, dual-host parity preserved, generated surfaces re-emitted (never hand-edited), `npm run verify` is the gate.

## Headline

- **84 raw findings** → **52 confirmed** (20 high-severity, 32 medium), **3 refuted**, **29 low-severity** carried along.
- Of confirmed: **7 very-high-leverage**, 25 high, 20 medium.
- **The codebase is fundamentally sound.** Write-side boundaries are well-guarded (zod-validated trace append, fail-closed run-file writes, byte-identical host bundles, allowlisted subprocess args). The problems are concentrated and *internal*: loose-typed seams that force hand-narrowing, projection/derivation logic duplicated across consumers, copy-pasted primitives, and a few god-functions.
- **Plan confidence per the completeness critic: MEDIUM.** Three blockers must be resolved before execution (see *Pre-execution hardening*).

## Dominant structural themes

- **Loose-typed seams force hand-rolled narrowing.** `ExecutableStep` and the runtime `TraceEntry` widen strict schemas to `unknown`/all-optional, so every executor and trace consumer re-asserts shape with `as` casts and ad-hoc guards. `RC-1`, `REP-1`, `REP-3`, `REP-4`, `RC-6`.
- **Projection / derivation duplicated across consumers.** The same facts are independently re-derived in multiple places with divergent tolerance: live vs cold run-status (`REP-2`), runtime vs emit-time stage/route/axis derivation (`GEN-1`), two staleness enumerators (`SD-1`). `RunClosedOutcome` is hand-re-enumerated in ~10 files.
- **Copy-pasted primitives never hoisted.** Canonical-JSON+SHA-256 hashing (`CS-1`/`CS-2`), test fixtures (`deterministicNow`×46, `RelayResult` stub in 41 files — `TEST-1`/`TEST-2`/`TEST-6`), route serialization four ways (`cli-4`), Commander bootstrap ×5 (`cli-5`).
- **God-functions fusing orthogonal concerns.** `graph-runner` `executeExecutableFlowOutcomeUnsafe` (~420 LOC) fuses recovery corridor + budget detection + proof-gap close + outcome binding (`RC-4`); CLI `main()` fuses dispatch/execution/output across 633 LOC (`cli-2`), with a triple-duplicated post-run block (`cli-1`).
- **Dependency-direction leaks.** `flows/` reaches up into `runtime/` (`flows-3`, `CH-3`); flow-specific projectors live in `shared/` and import flow code (`CS-3`).
- **Scattered config & hand-copied parity spine.** Built-in connector identity spread across 4+ files with no registry or exhaustiveness check (`CH-1`); the ~900/660-line host wrappers `plugins/{claude,codex}/scripts/circuit.ts` are maintained by hand-copy (`CH-2`).
- **Genuine latent bugs (not just style).** `closeRun` reads the primary-result file unguarded — a malformed/missing file can flip a completing run to `rejected` (`RC-3`); tests clear env via `process.env.X = undefined`, which sets the literal string `"undefined"` and can mask the real value (`TEST-3`).

## Per-subsystem assessment

### runtime-core-execution (src/runtime/run, src/runtime/run-files, src/runtime/executors, src/runtime/domain)

The run lifecycle and step-execution engine is functionally cohesive and well-guarded at its trace and run-file write boundaries (TraceStore.append re-parses every entry with the zod schema, RunFileStore.writeJson is fail-closed via the report registry). The dominant structural problems are: (1) executor step-policy fields (check/policy/join/concurrency) are typed as `unknown` on ExecutableStep, so all six executors re-assert their shape inline with unchecked `as {...}` casts and ad-hoc guards — a leaky manifest→executor abstraction that is the single highest-leverage harden+decomplect lever; (2) an in-progress RunValue/RunPorts "ports" abstraction (run-values.ts, 176 LOC) is wired into exactly ONE executor (compose) plus a contract test, leaving five executors on raw RunContext — a half-finished migration that adds surface without payoff; (3) graph-runner.ts (1059 LOC, executeExecutableFlowOutcomeUnsafe ~420 LOC) fuses the recovery-corridor state machine, cycle/attempt-budget detection, proof-gap close logic, and terminal-outcome binding into one giant loop; (4) read-side trust asymmetry — RunFileStore.readJson does `JSON.parse(raw) as T` with no validation, and closeRun's terminalOutcomeBoundToPrimaryResult reads the primary result unguarded so a malformed/missing primary-result file can flip a completing run to `rejected`; (5) pure helper duplication across executors (readRouteFromReport, proofIdPart/uniqueValues/proofAssessmentReportRef/stepCanCloseRun, errorFromUnknown, routeTargetKey, isRecord) and near-duplicate proof-assessment scaffolding between relay.ts and verification.ts; (6) `depth` is a loose `string` in RunContext while a strict Depth enum exists, and checkpoint.ts string-compares it without parsing (relay.ts does parse it). All findings are internal restructurings that preserve observable behavior, the trace/result contracts, flow taxonomy, and dual-host parity. Findings are ranked by leverage. Every change must keep `npm run verify` green (the runtime-context-boundary source-text contract test pins several current shapes, so some findings depend on relaxing that test in lockstep).

### runtime-evidence-projections

I deeply read all of src/runtime/{projections,trace,manifest,fanout,safe-apply}, src/run-status, and src/run-envelope. The subsystem is functionally solid (strong write-side zod discipline, a fail-closed append-only TraceStore, well-isolated pure run-envelope loop modules with good tests), but two structural problems dominate. (1) The loose runtime trace type src/runtime/domain/trace.ts deliberately widens the strict discriminated-union TraceEntry schema into an all-optional, string-widened bag. That single decision forces every trace consumer to hand-roll narrowing: progress.ts alone has ~10 re-narrowing helpers and 18 `as StepId`/`as ProgressRunId` casts, runtime-run-folder.ts re-parses the trace AND re-implements traceString/traceNumber/traceStringArray + the completed-step set logic, and the RunClosedOutcome enum (already a zod enum in trace-entry.ts) is re-enumerated by hand in at least 4 places. (2) Run status/folder projection is split across progress.ts (live, onAppend) and runtime-run-folder.ts (cold, from disk) which independently re-derive the same facts (outcome, current step, waiting checkpoint, tournament presentation) from the same trace with divergent tolerance — a classic projection-logic-duplicated-across-consumers smell. Beyond those: FanoutStep types branches/join/concurrency as `unknown` despite precise schemas existing, then branch-expansion re-casts them; the two long sibling fanout branch executors duplicate trace scaffolding; src/runtime/safe-apply is an empty orphan directory; and gitWorktreeRunner (real spawnSync) has zero test coverage. Highest-leverage move is to narrow domain/trace.ts toward the strict union (or add a single shared trace-narrowing/projection helper module), which collapses most of the duplicated narrowing in one shot. All findings are internal-only; no domain-model, command-surface, flow-taxonomy, or generated-surface changes.

### flows

The flows subsystem is a per-flow package model (8 flows: build, explore, fix, prototype, pursue, review + internal goal, runtime-proof) compiled by catalog.ts into CompiledFlowPackages that every downstream registry/router/emit derives from. The architecture is fundamentally sound — schemas are a clean leaf, the catalog is the single source of truth, and the writer/projection convention is consistent. The dominant issues are (1) per-flow boilerplate multiplied 7-8x: identical flow.ts/index.ts stubs (5 of 8 index.ts exports are dead but contract-pinned), near-byte-identical evidence_links de-dup superRefine blocks across build/fix/prototype (already factored in explore), and repeated relay-hint prose fragments; (2) two dependency-direction leaks — flows/prototype/writers/variant-options.ts reaches UP into runtime/connectors, and a misnamed close-writers/shared.ts re-export forces compose-writers and 9 writer files to import a general flow-graph helper through a close-specific module; (3) the router hardcodes flow-specific entry-mode regexes that partly duplicate each flow's declared routing signals; (4) the internal goal flow carries a full routing block (signals/order/reasonForMatch) that is dead — filtered out of classification by visibility:'internal' and excluded from the capabilities ledger; (5) one truly-dead function (classifyFlowStagePolicy). Findings are ranked by leverage. None change observable behavior or the domain model.

### connectors-hosts

The connectors-hosts subsystem is genuinely sound on its hardest seam: the dual-host runtime bundles (plugins/{claude,codex}/runtime/circuit.js) are byte-identical and generated from one esbuild output, git-state.ts is an emitted sidecar (not hand-editable), and the connector subprocess boundary is fail-closed (Codex argv allowlist, claude-code MCP/slash re-assertion at parse time, frozen-flag module-load invariants). The real architectural problems are concentrated in two places. (1) BUILTIN CONNECTOR METADATA IS SCATTERED: each built-in connector's identity (executable name, provider, supported efforts, filesystem capability, dispatch flags) is spread across the connector module, a hardcoded if-chain in runtime/connectors/resolver.ts (expectedProvider/supportedEfforts), the dispatch if-chain in runtime/executors/relay.ts, and schemas/connector.ts (BUILTIN_CONNECTOR_CAPABILITIES + EnabledConnector enum). Adding or editing a built-in connector requires synchronized edits across 4+ files with no single registry and no compile-time exhaustiveness check, and provider/effort assertions are duplicated between each connector file and the resolver. (2) THE HAND-WRITTEN HOST WRAPPERS (plugins/{claude,codex}/scripts/circuit.ts) are a 900+/660-line parity seam maintained entirely by hand-copy: resolveRuntimeCommand, version comparison, runtimeArgs/runtimeEnv, flow-root injection, and the doctor smoke scaffold are near-identical between hosts with deliberate divergences (Claude has the presentation/status-block renderer; Codex has codex-hooks checks) and zero shared module — divergence is caught only by separate host-plugin contract tests. Secondary findings: a flows->runtime dependency-direction violation (prototype writer imports the runtime connector resolver), a silent structured-output capability gap (cursor-agent and custom declare structured_output:'json' but ignore responseSchema), and near-duplicate NDJSON-parse/error-formatting scaffolding across connectors. All findings are internal restructuring only; none touch the flow taxonomy, command surface, connector behavior, or trace contracts.

### contracts-shared (src/schemas, src/types, src/shared incl. html, operator-summary)

The schemas/ layer is disciplined (Zod-first, .strict(), branded IDs, fail-closed superRefines) but carries two systemic seams: (1) the canonical-JSON-then-SHA256 hashing primitive is copy-pasted byte-for-byte across 4-6 files and produces durable ref.sha256 contract values, and (2) the 64-char-hex digest validator is independently re-derived 4-5 times (Sha256, ManifestHash, ContentHash, skill.sha256, history regex) instead of being a single shared brand. The shared/ layer is mostly coherent against its README rule but contains a real layering inversion: flow-specific HTML projectors and operator-summary projectors live under shared/ and import flows/*/reports.ts and flows/catalog.ts, plus a hardcoded write-capable flow-id set duplicated as a string literal. src/types/ is an empty seat (README only). The schemas/index.ts barrel is an underused 49-line wildcard hub (only 9 importers vs 24 direct). I found one near-duplicate derivation hazard (rubric dim-score/aggregate logic computed in shared/rubric.ts and re-validated by an identical superRefine in schemas/rubric.ts) that must stay byte-aligned or the schema rejects its own producer. Notably, the manifest_hash field is INTENTIONALLY polymorphic (runtime:flow@version pseudo-hashes appear in green contract tests), so it is NOT a Sha256-tightening target. Findings ranked by leverage; the hashing-helper and hex-brand consolidations are highest impact-to-effort and lowest risk.

### supporting-domains (src/history, src/process-evidence, src/skill-moments, src/release)

The four supporting domains are individually well-validated at their zod boundaries, but the history subsystem carries the highest-leverage defect: source-file enumeration for the staleness fingerprint is implemented twice with diverging rules (extract.ts's symlink-rejecting all-files walker vs indexer.ts's symlink-allowing .json-only walker), so the index can report possibly_stale immediately after a fresh rebuild whenever a non-JSON report file (run-surface.md, operator-summary.md/html) has the newest mtime. Secondary themes: (1) inconsistent persistence/serialization primitives across domains -- history hashes files as decoded utf8 strings while process-evidence and run-envelope hash raw bytes and run them through Sha256.parse, and sha256File/mtimeMs/runRelativePath are re-implemented per module; (2) branded-ID round-tripping (flow_id as unknown as string then re-parse) that re-validates already-branded values; (3) two supporting domains reach sideways into runtime/run-files/paths for a path-safety primitive that is foundational, not runtime-specific; (4) lossy non-exhaustive outcome projection in process-evidence; (5) the skill-moments domain is wired and contract-tested but has zero production callers, and its two decision-packet builders duplicate the same policy invariant the event builder already encodes; (6) release/checks compareTextAxis is a no-op (void key) that only presence-checks text axes, diverging from what the Parity-matrix domain term implies. All findings are internal restructurings that preserve observable behavior and the documented model; the staleness fingerprint fix is the one with an actual user-visible correctness improvement (eliminating spurious stale warnings) and must be guarded by a characterization test FIRST.

### generation-pipeline

The codegen/drift/release pipeline in scripts/ is functionally sound and fail-closed (emit.ts and runtime-bundle.ts both do exact-byte drift checks; the dual-host bundles are provably byte-identical). The structural problems are concentrated in three areas. (1) DECOMPLECT: scripts/release/emit-current-capabilities.ts re-derives flow facts (canonical stage order, executable route set, per-report semantic outputs, behavior axes) by re-parsing generated JSON and hand-maintaining lookup tables, instead of importing the runtime's own derivations that it already loads from dist/ -- the largest "scripts reimplement runtime logic" surface and the most drift-prone. (2) STREAMLINE: emit.ts encodes every artifact as a near-identical emit/check function pair (~9 such pairs), there are three separate biome-format subprocess wrappers, emit.ts spawns one npx biome process per generated file (40+ spawns per run), axisSelectionsFor/table/writeOrCheck are duplicated across release scripts, the four check-*.ts scripts are copy-paste twins, and scripts/release/shared.ts carries four dead helpers. (3) HARDEN/BOUNDARY: the dist-import seam erases all type information (loadReleaseChecks/loadReleaseSchemas return Promise<any>, forcing every release-script consumer to Record<string, any> even though src/release/checks.ts is fully typed); the Claude host-command renderer applies fragile chained regexes that silently no-op on source-prose drift with no fail-closed assertion (only the Codex skill path is guarded); and version.json is read and ad-hoc-shaped in four places. Every finding is internal-only and preserves observable bytes/behavior; the existing subprocess drift test (tests/unit/emit-flows-drift.test.ts) plus emit --check are the oracle.

### test-architecture

The test suite (213 files, ~73k LOC) is broad and the runtime engine/schemas are well-covered, but it carries heavy primitive-fixture duplication and a thin, underused tests/helpers/ layer. The single highest-leverage issue is that the most-copied test primitives — deterministicNow (46 files, byte-identical), the RelayResult stub literal (41 files), stdout/stderr capture (~19 files, 3 signatures), and withTempRun (3 defs) — were never hoisted into tests/helpers/, even though that hoist pattern already works for runtrace builders. Two correctness-grade HARDEN issues stand out: (1) several runner tests clear env vars via `process.env.X = undefined`, which coerces to the string "undefined" rather than deleting the key — masked for CIRCUIT_SHOW_RUNTIME_DECISION (checked `=== '1'`) but actively wrong for CIRCUIT_GENERATED_FLOW_MIRROR_ROOT (read via `?? process.env[...]`, so "undefined" becomes a truthy mirror-root path); the correct delete-based pattern exists in exactly one file. (2) config-loader.test.ts and build-checkpoint-exec.test.ts have copy-paste-drifted env save/restore blocks where multiple distinct `original*` vars all read/write the SAME key. Brittle-fixture coupling: emit-flows-drift.test.ts plants stale files directly into git-tracked generated/plugin dirs and runs `npm run build` + emit in-process, risking a dirty working tree if killed mid-run. The flow taxonomy is hardcoded as literal string arrays across 6 test files (with pursue/goal inconsistencies) rather than derived from the catalog. The two host-plugin contract tests duplicate helpers/schemas/constants verbatim. tests/parity/ is misnamed (per-flow runtime smoke, not dual-host parity) and its 885-line helper file is imported cross-suite. gitWorktreeRunner has zero direct coverage, blocking safe refactor of fanout worktree handling.
