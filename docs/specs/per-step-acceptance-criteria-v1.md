## Product Target

Add optional `acceptance_criteria` to relay steps so a step can prove its
claimed work before the flow advances.

The V1 target is deliberately narrow:

- deterministic checks only;
- criteria are authored with the step, not generated by the worker;
- hard-fail first;
- retry-with-feedback second;
- checkpoint escalation only after the checkpoint protocol can honestly support
  it.

The source idea frames the core gap as a relay step being accepted after the
worker returns, without a distinct validation check between worker return and
step completion. See
[docs/ideas/per-step-validation-check.md:83-120](../ideas/per-step-validation-check.md#L83-L120)
and
[docs/ideas/per-step-validation-check.md:200-238](../ideas/per-step-validation-check.md#L200-L238).

## Design Decisions

### 1. Inline typed criteria

Put `acceptance_criteria` inline on relay steps, not in a side file.

Why:

- the source idea recommends step-authored criteria and leans inline;
- current schematic and compiled step schemas are already the contract surface;
- discoverability matters more than compactness for V1.

Rejected option: a separate criteria registry. That may help later if criteria
are reused heavily, but it adds a second lookup path before the basic behavior
has proved itself.

### 2. Deterministic V1 kinds only

Start with two criterion kinds:

```ts
type AcceptanceCriterion =
  | {
      kind: 'command';
      id: string;
      command: VerificationCommand;
      expected_status: 'passed';
    }
  | {
      kind: 'report_field';
      id: string;
      path: readonly string[];
      predicate: 'present' | 'non_empty';
    };
```

`command` reuses verification command semantics. `report_field` checks the
already-parsed relay report and stays deliberately small.

Rejected option: `schema` as a broad criterion kind. The relay executor already
schema-validates the whole report, and a broad `schema` label would blur report
field checks with external file checks.

Rejected option: LLM-judged criteria. The source idea explicitly warns that a
validator using the same prompt loop as the worker does not solve the problem.

### 3. Use existing recovery attempts

Do not create a second retry counter inside `acceptance_criteria`.

If a relay step wants three repair attempts, author or compile
`budgets.max_attempts: 3` on that step. Current retry exhaustion tests already
assert the default 2-attempt recovery behavior. See
[tests/parity/fix.test.ts:201-226](../../tests/parity/fix.test.ts#L201-L226).

### 4. Checkpoint escalation is a later protocol

V1 hard-fail and retry-with-feedback can be honest without new operator actions.
Checkpoint escalation should wait until there is an authored checkpoint step or
a saved-manifest migration protocol.

Do not claim support for "edit criterion and retry" through current checkpoint
resume. Current resume validates the saved manifest and request hash; changing a
criterion means changing the saved flow definition.

## Implementation Slices

### Slice 1 - Contract Shape, No Runtime Behavior

Goal: accept and preserve optional relay-step criteria without running them.

Owner files:

- `src/schemas/acceptance-criteria.ts` - new schema module.
- `src/schemas/step.ts` - add optional `acceptance_criteria` to `RelayStep`
  only.
- `src/schemas/flow-schematic.ts` - add optional schematic field and reject it
  on non-relay execution kinds.
- `src/flows/compile-schematic-to-flow.ts` - pass the field through in the relay
  branch.
- `src/runtime/manifest/executable-flow.ts`,
  `src/runtime/manifest/from-compiled-flow.ts`,
  `src/runtime/manifest/runtime-package-index.ts`, and
  `src/flows/registries/runtime-index.ts` - preserve the field through runtime
  projection.
- `src/flows/block-step-expansion.ts` - allow flow data to author
  `acceptanceCriteria` and pass it through as `acceptance_criteria`.
- `docs/contracts/step.md` and `docs/contracts/compiled-flow.md` - document the
  optional contract field and its compatibility story.

Expected behavior:

- existing flows with no criteria behave byte-for-byte the same except for any
  intentional generated metadata changes;
- no relay execution path runs criteria yet.

Verification:

### Slice 3 - Deterministic Validator, Hard-Fail Only

Goal: run criteria after relay acceptance and before step completion.

Runtime plan:

1. Keep the existing relay verdict, schema parse, and cross-report validation
   order.
2. If those pass, run `acceptance_criteria`.
3. If all criteria pass, keep the existing route and report-writing behavior.
4. If a criterion fails, emit validation trace, do not write the canonical
   report, and fail the step cleanly.

Trace plan:

- Extend `check.evaluated.check_kind` to include `acceptance_criteria`, or add a
  dedicated `acceptance_criteria.evaluated` trace entry. Prefer extending
  `check.evaluated` only if criterion id/kind/output summary can be represented
  without making the event ambiguous.
- Record criterion id, criterion kind, pass/fail, reason, and bounded command
  output summaries when applicable.

Test plan:

- passing criteria do not change no-criteria trace order;
- failing report-field criterion aborts before `step.completed`;
- failing command criterion records bounded stdout/stderr summaries;
- fail-closed behavior holds when criteria reference a missing report field;
- existing schema-parse rejection behavior still holds.

Verification:

```bash
npm run test -- tests/contracts/runtrace-schema.test.ts tests/runner/materializer-schema-parse.test.ts tests/runtime/control-loop.test.ts
npm run test -- tests/runner/build-runtime-wiring.test.ts
npm run verify:fast
```

Stop conditions:

- relay steps without criteria emit new trace entries;
- a criteria failure writes the canonical typed report as if the step passed;
- command criteria can escape project root or run through a shell;
- `TraceStore` accepts untyped validation history.

### Slice 4 - Retry With Feedback

Goal: when a criterion fails and the step declares retry behavior, the next
relay attempt receives the failed criterion as focused feedback.

Runtime plan:

- Route failed criteria through existing `retry` recovery when the step declares
  a retry route.
- Use the graph runner's attempt bounds instead of a separate retry counter.
- Add a compact feedback section to the next relay prompt. It should include the
  failed criterion id, criterion kind, reason, and bounded output summary.

Repo seam:

- `RunContext` currently exposes `activeStepAttempt` but not active recovery
  details. See
  [src/runtime/run/run-context.ts:10-30](../../src/runtime/run/run-context.ts#L10-L30).
- `composeRelayPrompt` currently accepts only the step, run folder, and loaded
  skills. See
  [src/shared/relay-support.ts:80-107](../../src/shared/relay-support.ts#L80-L107).

Viable options:

1. Pass explicit recovery feedback through `RunContext`.
2. Reconstruct the latest failed criterion from trace inside the relay executor.
3. Write a small run-relative feedback report and add it to `reads`.

Preferred option: explicit `RunContext` feedback. It is testable and avoids
making prompt composition depend on scanning trace history.

Verification:

```bash
npm run test -- tests/runtime/control-loop.test.ts tests/runner/build-runtime-wiring.test.ts tests/runner/check-evaluation.test.ts
npm run verify:fast
```

Stop conditions:

- a step declares retry-with-feedback but has no `retry` route;
- retry prompt omits the failed criterion reason;
- retry attempts exceed `budgets.max_attempts`;
- feedback appears on first attempts or on unrelated relay steps.

### Slice 5 - Checkpoint Escalation

Goal: let the operator decide after validation retries are exhausted.

Do not synthesize an invisible checkpoint inside the relay executor. Prefer an
authored checkpoint step that reads a validation-failure report and uses current
checkpoint mechanics.

Minimum honest operator choices:

- retry after the operator manually edits the worktree or output;
- stop;
- handoff or escalate.

Deferred:

- edit criterion and retry;
- edit output inside the checkpoint response;
- manifest mutation during resume.

Verification:

```bash
npm run test -- tests/runtime/checkpoint-resume.test.ts tests/runner/build-checkpoint-exec.test.ts tests/runtime/control-loop.test.ts
npm run check-release-infra
npm run verify
```

Stop conditions:

- checkpoint resume accepts changed criteria without an explicit saved-manifest
  migration story;
- relay-created pauses bypass checkpoint request hashing;
- operator choices are not represented as normal routes.

## Rollout Risks

- Public contract drift: optional fields are still public fields because
  compiled flow manifests are host-visible generated outputs.
- Trace noise: validation history is useful only if it stays compact and
  criterion-keyed.
- False confidence: report-field checks are not a replacement for full
  verification or Review.
- Retry loops: use existing attempt bounds; do not add an unbounded validator
  loop.
- Checkpoint overreach: editing criteria is a flow-definition mutation, not a
  normal checkpoint selection.

## Claim Inventory

Confirmed:

- `acceptance_criteria` is optional and relay-only;
- V1 criterion kinds are deterministic `report_field` and `command`;
- command criteria reuse existing direct-argv proof command constraints;
- generated surfaces must be regenerated from sources when flow data changes;
- recovery default is 2 attempts for retry/revise routes unless
  `budgets.max_attempts` overrides it.

Supported inference:

- keeping `schema_version: "2"` remains acceptable because the field is
  backward-compatible and optional, and generated runtime/manifests ship
  together.

False or stale:

- "default 3 retries" is not current behavior;
- "no current `acceptance_criteria` field exists on relay steps" is stale.

Blocked for V1:

- "edit criterion and retry" through checkpoint resume.
